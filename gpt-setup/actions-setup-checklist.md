# Actions Setup Checklist

Use this checklist to connect the Partner Intake OS Custom GPT to the Vercel API through the Batch 07 OpenAPI Action Pack.

## Prerequisites

Before adding Actions, confirm:

| Check | Requirement |
|---:|---|
| [ ] | Batch 05 API scaffold is deployed or ready to deploy. |
| [ ] | Batch 06 storage mode is set to `mock`, `json`, or a configured connector. |
| [ ] | Batch 07 `/actions/openapi.yaml` exists. |
| [ ] | Vercel project domain is known. |
| [ ] | `PARTNER_INTAKE_ACTION_TOKEN` exists in Vercel. |
| [ ] | Health endpoint returns a response. |

## Required Vercel env var

Set this in Vercel:

```txt
PARTNER_INTAKE_ACTION_TOKEN=replace_with_long_random_token
```

Recommended token style:

```txt
pio_live_64plus_random_chars_here
```

Do not use:

```txt
TALLY_SIGNING_SECRET
```

That secret is for Tally webhook verification, not GPT Actions.

## OpenAPI file

Use:

```txt
/actions/openapi.yaml
```

Optional alternate format:

```txt
/actions/openapi.json
```

## Replace server URL

Inside `openapi.yaml`, replace:

```txt
https://YOUR_VERCEL_DOMAIN.vercel.app
```

with the live API domain.

Example:

```txt
https://partner-command-center.vercel.app
```

## Import OpenAPI YAML

In the GPT editor:

1. Open the Partner Intake OS GPT editor.
2. Go to **Configure**.
3. Go to **Actions**.
4. Select **Create new action**.
5. Import or paste:

```txt
/actions/openapi.yaml
```

6. Confirm detected actions appear.

Expected operation IDs:

```txt
checkHealth
classifyPartnerIntake
recommendPartnerResources
generatePartnerOnboardingPlan
generatePartnerCampaignKit
logPartnerEvent
```

## Set Bearer auth

Set authentication to:

```txt
Authentication type: API key
API key placement: Bearer
Header generated by GPT: Authorization
```

## Paste token

Paste the same token used in Vercel:

```txt
Bearer token value: YOUR_PARTNER_INTAKE_OS_ACTION_TOKEN
```

Expected outbound header:

```txt
Authorization: Bearer YOUR_PARTNER_INTAKE_OS_ACTION_TOKEN
```

## Endpoint allowlist

The Action Pack should expose only:

```txt
GET  /api/health
POST /api/partners/classify
POST /api/partners/recommend-resources
POST /api/partners/generate-onboarding-plan
POST /api/partners/generate-campaign-kit
POST /api/partners/log-event
```

It must not expose:

```txt
POST /api/tally/partner-intake-webhook
```

## Test endpoints

### Test 1 — Health check

Prompt:

```txt
Use the Partner Intake OS API to check system health.
```

Expected result:

- Action called: `checkHealth`
- Response includes system status
- No auth error

### Test 2 — Classify partner intake

Prompt:

```txt
Use the Partner Intake OS API to classify this partner intake:

First name: Dana
Last name: Brooks
Email: dana@example.com
Company: Books & Backoffice Co.
Partner type claimed: CPA/bookkeeper
Audience: contractors and service businesses
Funding experience: refers clients who need working capital education
Referral volume estimate: 2-5 per month
Desired role: referral partner
Source: manual
```

Expected result:

- Action called: `classifyPartnerIntake`
- Partner type should be close to `cpa_bookkeeper`
- Risk should likely be `low` or `medium`
- Manual review should be false unless missing data or risk flags appear
- Output should include next action

### Test 3 — Recommend resources

Prompt:

```txt
Use the Partner Intake OS API to recommend resources for a CPA/bookkeeper partner serving contractors and local service businesses.
```

Expected result:

- Action called: `recommendPartnerResources`
- Recommendations should include business funding readiness, referral scripts, Funding Product Matrix, or Partner Command Center-style resources
- No guaranteed funding language

### Test 4 — Generate onboarding plan

Prompt:

```txt
Use the Partner Intake OS API to generate an onboarding plan for a Tier 2 referral partner on the standard affiliate partner path.
```

Expected result:

- Action called: `generatePartnerOnboardingPlan`
- Includes first 24 hours
- Includes first 7 days
- Includes first 30 days
- Includes owner and next action

### Test 5 — Generate campaign kit

Prompt:

```txt
Use the Partner Intake OS API to generate a campaign kit for a creator affiliate serving gig workers and early-stage founders.
```

Expected result:

- Action called: `generatePartnerCampaignKit`
- Includes campaign angle, CTA, suggested channels, lead magnet, and compliance guardrails
- No fake urgency or invented testimonials

### Test 6 — Log event

Prompt:

```txt
Use the Partner Intake OS API to log that this partner was reviewed and needs manual follow-up.
```

Expected result:

- Action called: `logPartnerEvent`
- Mock logged status is acceptable for MVP
- No live CRM write is required unless Batch 06 storage is configured

## Verify operation IDs

Confirm the GPT editor detects:

```txt
checkHealth
classifyPartnerIntake
recommendPartnerResources
generatePartnerOnboardingPlan
generatePartnerCampaignKit
logPartnerEvent
```

If one is missing, inspect `actions/openapi.yaml` before testing.

## Bad auth test

Temporarily use the wrong token in the GPT Action auth settings.

Expected result:

- Auth fails
- Protected partner endpoints reject the request
- Restore the correct token immediately after test

## Troubleshooting

### The action does not appear

Check:

- YAML indentation
- OpenAPI version
- server URL
- operation IDs
- schema validity

### Auth fails

Check:

- GPT Action token exactly matches Vercel `PARTNER_INTAKE_ACTION_TOKEN`
- no extra spaces
- Bearer auth selected
- Vercel deployment has latest env vars

### Endpoint returns 404

Check:

- Vercel deployment route structure
- API file path
- domain replacement in OpenAPI server URL
- production deployment is current

### GPT calls the wrong action

Tighten `/gpt/instructions.md` and add explicit action-use rules. Do not add more endpoints to fix a routing problem. That is how spaghetti learns to drive.
